# 回答一

**公司/项目/时间/角色：** 华为海思视觉芯片解决方案研发部，AI代码扫描系统，2025.09至今，项目负责人+Agent开发

## 项目简介

为了扫描芯片C语言代码中的**网络安全问题**和**代码逻辑问题**，研发的一套AI代码全量扫描工具，包含前端扫描页面、前端标注页面、后端管理系统、后端Agent核心。

## 我负责的

整体系统设计、后端Agent核心开发

## 项目核心方案

1. 通过AST方式将代码中函数、全局变量、结构体、宏定义信息，通过映射表的方式存储起来，开放function工具能通过函数名/变量名来获取具体实现。
2. 大模型能通过FunctionCall方式自主按需获取代码上下文信息，通过Re-Act循环来深入探索上下文，从而发现隐藏在调用链中的深层次问题。

## 遇到的输出不可控问题和解决方案

### 问题1：大模型有时可以找到一个问题，有时又无法找到

大模型Attention机制下注意力不集中导致的问题。解决方案：

- 将任务深度拆解，在资源和性能允许情况下尽量拆解成最小子任务
- 本项目中拆成每个函数进行一次单独扫描，每次扫描有6个细分维度：网络安全问题、资源类问题、校验类问题、数值类问题、逻辑类问题、编码规范问题
- 即每个函数都会经历6次子扫描
- 部分问题可能分类模糊，最后需进行一次去重

### 问题2：大模型输出JSON格式不稳定

两种解决方案：

- **方案一：** 放弃JSON格式，使用文本Markdown格式让大模型输出，用正则去提取。大模型输出Markdown的稳定性非常好
- **方案二：** 通过Python进行容错处理：
  1. 先通过正则匹配JSON符号，去掉不相干的文字
  2. 替换中文字符/错误字符为英文（如双引号、单引号）
  3. 如果仍无法解析，插入上下文把报错堆栈发给大模型，让其修正后再次输出JSON

### 问题3：大模型调用工具不稳定

起初的方案有多个function函数（查询函数实现、结构体实现等），过程中发现大模型幻觉导致会用错工具、分不清查询类型，或把函数名写多/写错。解决方案：

- 全部融合成一个工具，可查询函数、结构体、变量等
- 对入参进行清洗：去掉多余字符、空格，大小写优化等容错
- 入参改为list列表，一次调用可查多个函数实现，缩短查询次数

### 问题4：大模型输出幻觉，内容错误

单个Agent扫描总会出现幻觉问题，解决方案：

- 加一个检视Agent：扫描Agent扫出问题后，检视Agent针对每个问题进行推理验证
- 经过推理验证的问题才会被采纳，输出到最终用户界面
- 设置置信度分数、严重程度指标，仅置信度高于阈值或严重问题才输出到用户界面

# 回答二

## 场景构造

先构造测试场景和日志输入。选用开源的法律合规检查Agent（内有多个检查子Agent）：

- 项目地址：<https://github.com/zubair-trabzada/ai-legal-claude>
- 分别用 `glm5.1` 和 `minimax-m2.7` 通过 Claude Code 对文章 [`sample/docx/006776_华夏鼎略债券型证券投资基金招募说明书.md`](sample/docx/006776_华夏鼎略债券型证券投资基金招募说明书.md) 进行合规检查测试
- 将 `.claude/projects/` 里的JSON对话日志作为测试输入：[`sample/log`](sample/log)

## 功能设计
*以下是人工撰写的*
### 输入

两份日志：`sample/log/log_glm5.1_20260505`（基线）和 `sample/log/log_minimax2.7_20260505`（升级后）。后者为模型升级后的日志，包含从Claude Code导出的JSON日志：

- 根目录有一个JSON文件：主Agent的对话及工具调用日志
- 内含一个子目录：子Agent日志

### 输出

一份清晰的HTML报告，包含：

- 两份日志的多项指标数据（主/子Agent分别统计）
- 整体对比情况
- 退化风险判断和处理建议

### 整体方案

#### 统计指标

用Python实现，各指标检测规则由单独类管理，继承自基类，保证可插拔可扩展：

| 类别 | 指标 | 理由 |
|------|------|------|
| 资源消耗类 | token使用量、Agent总会话时长、Agent平均每秒token数、会话执行总步骤数 | 评估费用成本、响应性能和上下文精度损耗 |
| 调用工具类 | 工具调用步骤、工具调用失败次数/失败占比、主Agent子Agent调用数量 | 工具调用过少代表依赖空想，过多代表决策混乱 |
| 深度思考类 | think的token总数、think次数 | 异常升高代表推理冗余，降低代表思考不足 |
| 内容质量类 | 模糊不确定性词汇（可能、大概、或许、建议、疑似等）命中次数、单条句子完全重复出现3次以上次数 | 命中越多代表模型答案模糊、专业能力下降 |

**以上是业务指标，其他的需要根据业务实际情况来制定，比如输出内容的checklist、多次调用的结果稳定性对比，实际业务上必须结合专业评测集来衡量召回率、精确率**
#### 退化风险判断

通过大模型对比来实现：使用 `claude -p "提示词"` 方式，将提示词和cmd命令下发流程写到Python代码中，提示词中把客观数据对比结果和原始log路径都给到Claude，让其进行对比并给出退化风险判断和处理建议。（AI主观分析，可能还要细挑提示词工厂或者做成agent，根据不同业务制定不同的规则，当前项目只是简单的模拟）


经过Superpower多轮交互后给出的设计：[`docs/superpowers/specs/2026-05-05-agent-log-checker-design.md`](docs/superpowers/specs/2026-05-05-agent-log-checker-design.md)

## 工具使用方式

### 环境要求

- Python 3.10+
- [Claude Code CLI](https://docs.anthropic.com/en/docs/claude-code)（用于 LLM 退化风险判断，可选）

### 快速开始

```bash
# 完整运行（含LLM判断）
python main.py \
  --baseline sample/log/log_glm5.1_20260505 \
  --upgraded sample/log/log_minimax2.7_20260505 \
  --output output/report.html

# 跳过LLM判断（仅生成指标对比报告，若output/下有缓存的analysis.md会自动加载）
python main.py \
  --baseline sample/log/log_glm5.1_20260505 \
  --upgraded sample/log/log_minimax2.7_20260505 \
  --output output/report.html \
  --skip-llm
```

### 参数说明

| 参数 | 必填 | 默认值 | 说明 |
|------|------|--------|------|
| `--baseline` | 是 | - | Baseline 日志目录路径 |
| `--upgraded` | 是 | - | Upgraded 日志目录路径 |
| `--output` | 否 | `output/report.html` | 输出 HTML 报告路径 |
| `--skip-llm` | 否 | - | 跳过 LLM 退化风险判断步骤 |

### 输入日志格式

兼容 Claude Code 导出的 JSONL 日志结构：

```
log_dir/
├── <session-id>.jsonl          # 主 Agent 对话日志
└── subagents/
    ├── <agent-id>.jsonl        # 子 Agent 对话日志
    └── <agent-id>.meta.json    # 子 Agent 元信息
```

### 输出文件

所有输出统一存放在 `output/` 目录下：

| 文件 | 说明 |
|------|------|
| `report.html` | HTML 可视化对比报告 |
| `analysis.md` | LLM 生成的退化风险分析报告（Markdown 格式） |

### 执行流程

```
[1/4] 解析日志 → JSONL 解析为主/子 Agent 会话结构
[2/4] 计算指标 → 4 类 14 项指标（资源/工具/思考/内容）
[3/4] LLM 判断 → 调用 claude -p 进行退化风险分析（可跳过）
[4/4] 生成报告 → 输出自包含 HTML 报告
```

### 容错机制

- LLM 调用失败时自动重试（默认 2 次），超时 600 秒
- 所有重试失败后自动加载 `output/analysis.md` 缓存
- `--skip-llm` 模式下也会尝试加载缓存分析
- 多路径查找 `claude` 命令（PATH / npm 全局安装 / npx 回退）

### 扩展指标

指标通过继承 `BaseMetric` 基类实现插拔式扩展：

```python
# src/metrics/base.py
class BaseMetric(ABC):
    name: str          # 指标英文名
    category: str      # 所属类别 (resource/tool/thinking/content)

    @abstractmethod
    def compute(self, session: LogSession) -> dict:
        """计算指标，返回 {category: {key: value}} 结构"""
```

新增指标只需：1) 创建类继承 `BaseMetric`，2) 在 `main.py` 中 `pipeline.register(YourMetric())`
